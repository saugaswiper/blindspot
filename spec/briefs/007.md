# Brief 007 — Fix Remaining `AND NOT systematic[sb]` in PubMed Sibling Functions

**Date:** 2026-06-24
**Stage in pipeline:** Stage 2 — Literature search (primary-literature capture) + Stage 3 — Screening feed
**Status:** UNBLOCKED — pure code fix, no ops/secrets dependency

---

## Goal

Apply the binary `NOT` fix (proven in handoff 106, validated in validation 006) to the three
remaining call sites in `lib/pubmed.ts` that still use `AND NOT systematic[sb]`. These functions
currently count and fetch systematic reviews when they are supposed to exclude them.

---

## Why now

Validation 006 §5 F4 (Medium) confirmed three sibling functions carry the same ATM-inversion
defect fixed in handoff 106:

| Function | Line | Impact |
|----------|------|--------|
| `countPrimaryStudies` | 124 | Feasibility scoring display shows SR count, not primary study count |
| `fetchPrimaryStudiesForScreening` | 234 | **Screening workbench receives systematic reviews** — users screen wrong record type |
| `countPrimaryStudiesRecent` | 293 | Study-count trend (NEW-2) shows SR growth, not primary study growth |

Brief 006 explicitly specified all four sites (Sites 1, 2, 4). Handoff 106 only implemented
Site 3 (`fetchPrimaryStudyIds`, line 276). These three remain.

`fetchPrimaryStudiesForScreening` is the highest-priority: the screening workbench is the product
beachhead, and it is currently operating on systematic reviews. A user who runs screening on any
PubMed query gets SRs to screen, not the primary studies the review is supposed to evaluate.

`countPrimaryStudies` feeds both the live feasibility display and the cache
(`setCachedTopicCounts`). Cached values written under the bug persist until evicted, so the fix
also corrects cache writes going forward.

Per planner rules, a verified correctness bug on a shipped user-facing stage outranks new
features.

## Exact files to touch

| File | Lines | Action |
|------|-------|--------|
| `lib/pubmed.ts` | 124, 234, 293 | Replace `AND NOT systematic[sb]` → `NOT systematic[sb]` (binary NOT). No other changes. |

No changes to `lib/europepmc.ts`, `lib/openalex.ts`, `app/`, `components/`, `supabase/`,
or any other file.

### The three substitutions

Use the same binary NOT form proven by handoff 106 and verified by validation 006 (4/5 recall;
`AND NOT` and `NOT` result sets fully disjoint — 0 overlap — confirming binary NOT is honoured).

**Site 1 — `countPrimaryStudies` (line 124)**

```ts
// Before:
const { count } = await esearch(`(${query}) AND NOT systematic[sb]${datePart}`, 1);

// After:
const { count } = await esearch(`(${query}) NOT systematic[sb]${datePart}`, 1);
```

**Site 2 — `fetchPrimaryStudiesForScreening` (line 234)**

```ts
// Before:
const { ids } = await esearch(`(${query}) AND NOT systematic[sb]`, Math.min(limit, 10000));

// After:
const { ids } = await esearch(`(${query}) NOT systematic[sb]`, Math.min(limit, 10000));
```

**Site 3 — `countPrimaryStudiesRecent` (line 293)**

```ts
// Before:
url.searchParams.set("term", `(${query}) AND NOT systematic[sb]`);

// After:
url.searchParams.set("term", `(${query}) NOT systematic[sb]`);
```

Note: `countPrimaryStudiesRecent` constructs the URL directly rather than calling `esearch()`.
The substitution is the same filter string; only the construction path differs.

## Notes for the dev

**Why binary NOT works:** PubMed parses `(topic) AND NOT systematic[sb]` as `(topic) AND
[systematic]` due to ATM. Binary `NOT` — i.e., `(topic) NOT systematic[sb]` — is handled as
a boolean set-difference operation, which ATM cannot silently convert. This is the same
mechanism validated in handoff 106 / validation 006 probe B2 (disjoint sets, 0 overlap).

**Cache side-effect:** After fixing Site 1 (`countPrimaryStudies`), the next call for any
query that was previously cached under the wrong (SR) count will miss the cache (the old
value persists until TTL), re-fetch from PubMed, and write the corrected primary-study count.
No explicit cache-invalidation step is needed; this is the correct behaviour.

**Post-fix validation:** The tester will write validation 007 targeting these three functions.
Expected outcome: `countPrimaryStudies` and `countPrimaryStudiesRecent` return counts consistent
with primary studies (thousands, not ~313); `fetchPrimaryStudiesForScreening` does not return SR
PMIDs. The tester will also add a second SR truth set (validation 006 §8 recommended step 5)
to check generalisation.
