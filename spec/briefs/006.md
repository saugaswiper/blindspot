# Brief 006 — Fix PubMed Systematic-Review Exclusion Filter

**Date:** 2026-06-24
**Stage in pipeline:** Stage 2 — Literature search (primary-literature capture)
**Status:** UNBLOCKED — pure code fix, no ops/secrets dependency

---

## Goal

Replace every occurrence of `AND NOT systematic[sb]` in `lib/pubmed.ts` with an explicit
publication-type exclusion that PubMed's Automatic Term Mapping (ATM) cannot silently invert.
This is the root-cause fix for validation 005 F1 (CRITICAL): the function named
`fetchPrimaryStudyIds` currently returns **systematic review IDs**, not primary study IDs.

---

## Why now

Validation 005 (2026-06-23) confirmed that `AND NOT systematic[sb]` and `AND systematic[sb]`
return **identical result sets** (313/313 overlap, 100%) for the Mitchell 2012 test query.
PubMed ATM translates the query as:

```
Query sent:   (cognitive behavioral therapy insomnia) AND NOT systematic[sb]
ATM output:   (...) AND "systematic"[Filter]          ← NOT is dropped
```

The `NOT` is silently discarded, leaving a positive systematic-review filter. Every call that
uses `AND NOT systematic[sb]` is therefore:

- `fetchPrimaryStudyIds` (line 268): returns SR IDs for the recall pipeline → **0% recall on Mitchell 2012 truth set; production FAIL**
- `fetchPrimaryStudiesForScreening` (line 234): returns SR titles/abstracts to the screening workbench → screening operates on wrong record type
- `countPrimaryStudies` (line 124): counts SRs instead of primary studies → dashboard count is wrong
- `countPrimaryStudiesRecent` (line 285): same — recent primary-study count is wrong

Handoff 105 (relevance sort + 2000-record ceiling) correctly implemented brief 005 but could
not improve recall because the filter failure precedes any truncation. The existing pagination
fix is sound and stays; only the filter string changes.

Per planner rules a measured CRITICAL FAIL on a shipped stage outranks all new features.

## Exact files to touch

| File | Action | What changes |
|------|--------|--------------|
| `lib/pubmed.ts` | **EDIT** | Replace `AND NOT systematic[sb]` → `NOT (systematic review[pt] OR meta-analysis[pt])` at all 4 call sites (see below). No structural changes — same function signatures, same callers, same logic. |

No changes to `lib/europepmc.ts`, `lib/openalex.ts`, `app/`, `components/`, `supabase/`,
or any other file. EuropePMC's `NOT PUB_TYPE:"Systematic Review"` filter is a separate syntax

### The four substitutions

The replacement filter is:
```
NOT (systematic review[pt] OR meta-analysis[pt])
```

`[pt]` is PubMed's Publication Type field tag. It is a controlled vocabulary lookup, not a
free-text term, so ATM cannot expand or reinterpret it. "Systematic Review" and "Meta-Analysis"
are both valid PubMed MeSH publication types. This is the standard approach used in published
MEDLINE search strategies.

**Site 1 — `countPrimaryStudies` (~line 124)**

```ts
// Before:
const { count } = await esearch(`(${query}) AND NOT systematic[sb]${datePart}`, 1);

// After:
const { count } = await esearch(`(${query}) NOT (systematic review[pt] OR meta-analysis[pt])${datePart}`, 1);
```

**Site 2 — `fetchPrimaryStudiesForScreening` (~line 234)**

```ts
// Before:
const { ids } = await esearch(`(${query}) AND NOT systematic[sb]`, Math.min(limit, 10000));

// After:
const { ids } = await esearch(`(${query}) NOT (systematic review[pt] OR meta-analysis[pt])`, Math.min(limit, 10000));
```

**Site 3 — `fetchPrimaryStudyIds` (~line 268)**

```ts
// Before:
const { ids } = await esearch(`(${query}) AND NOT systematic[sb]${datePart}`, limit, "relevance");

// After:
const { ids } = await esearch(`(${query}) NOT (systematic review[pt] OR meta-analysis[pt])${datePart}`, limit, "relevance");
```

**Site 4 — `countPrimaryStudiesRecent` (~line 285)**

```ts
// Before:
url.searchParams.set("term", `(${query}) AND NOT systematic[sb]`);

// After:
url.searchParams.set("term", `(${query}) NOT (systematic review[pt] OR meta-analysis[pt])`);
```

Note: Site 4 builds the URL directly (not via `esearch()`). The substitution is the same
filter string; only the construction path differs.

## Notes for the dev

**Why `[pt]` beats `[sb]`:** PubMed's SubsetBank `[sb]` (e.g., `systematic[sb]`) is a
pre-computed filter maintained by NLM that is subject to ATM expansion. The `[pt]` (Publication
Type) field tag is a direct controlled-vocabulary lookup against the MEDLINE record's PT field,
which ATM does not expand. The standard literature-search guidance (Cochrane, JBI, PRISMA-S)
recommends explicit PT exclusion for robust search strategies.

**Boolean AND is implicit before NOT:** PubMed interprets `(query) NOT filter` as
`(query) AND NOT filter`, so the `AND` before `NOT` in the original string is not required.
Both `(query) AND NOT ...` and `(query) NOT ...` are equivalent. The replacement uses the
shorter form for clarity.

**`searchExistingReviews` must stay untouched:** That function (line 98) uses
`AND systematic[sb]` as a positive filter to *find* SRs for the existing-review display.
This is intentional and correct; the ATM translation into `AND "systematic"[Filter]` is
acceptable there because we *want* SRs returned.

**Post-fix validation:** after shipping, re-run validation harness with the same Mitchell 2012
truth set. The tester will write validation 006. Expected outcome: PubMed returns ≥4/5 truth
papers (all except McCluskey 1991 which does not use CBT terminology). If PubMed still returns
0/5, check the ATM translation of the new query using the NCBI ESearch `usehistory` flag or
the PubMed web interface to confirm `[pt]` is not being rewritten.
