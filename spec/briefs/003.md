# Brief 003 — Full-Text Retrieval: UI Surface and Persistence

**Date:** 2026-06-23
**Stage in pipeline:** 3 (full-text retrieval) — close-out pass, items 2 & 4 of brief 002
**Status:** UNBLOCKED — no ops/secrets dependency

---

## Goal

Surface the full-text resolution result to users in the screening workbench and make it
durable: add a "Get full text" button on `include`-verdict study rows, replace it with a
labeled source chip once resolved, and persist the resolved URL/source to the database so it
survives page reload.

---

## Why now

Brief 002 shipped four items. One (AC2 latency, concurrent fanout) was verified by
validation 003. The remaining three are:

| Item | Status |
|------|--------|
| AC2 latency fix | ✅ Verified (validation 003, p95 = 926 ms) |
| **AC5 UI surface** | ❌ Not started — zero fulltext UI in `ResultsDashboard.tsx` |
| **AC6 upload fallback** | Deferred → brief 004 |
| **AC-persist persistence** | ❌ Not started — `screening/save` route ignores fulltext fields |

`app/api/fulltext/route.ts` is live and tested, but a user who screens a study to `include`
today has no way to reach it. The resolution chain is entirely invisible.

Data extraction (the next pipeline milestone, Otto-SR parity step 4) is gated on full text
being **surfaced and persisted**. Building extraction on top of a feature with no UI and no
durable state would create hidden, unreliable infrastructure. Closing AC5 + AC-persist is the
minimum viable foundation for stage 4.

AC6 (upload fallback for paywalled studies) is intentionally deferred to brief 004. It
requires a Supabase Storage bucket and is separable from the core button flow.

## Scope

### In
- Migration: `fulltext_url`, `fulltext_source`, `fulltext_fetched_at` columns on
  `screening_results`
- Persistence: save the three fields via the existing `screening/save` upsert path when a
  resolution result is present
- UI: "Get full text" button on `include`-verdict rows → spinner → source chip (success) or
  "No open-access version found" note (404)
- Hydration: rows with a saved `fulltext_url` render the source chip immediately on page load
  without re-fetching

### Out
- AC6 / upload fallback — brief 004 (requires Supabase Storage bucket setup)
- Data extraction or any LLM processing of the PDF content — brief 005+
- Bulk background pre-fetch for all included studies — on-demand only
- Full-text screening (using full text to re-evaluate inclusion) — a distinct future stage
- Signed or access-controlled Supabase Storage URLs — not yet needed (no upload yet)
- CRIT-1 fix (`OPENALEX_API_KEY` on Vercel) — ops blocker, not in dev scope

## Acceptance criteria — beat, don't match

Otto-SR fetches open-access PDFs automatically but shows no source provenance. The bar: match
the capability, beat it on **auditable transparency** — every resolved URL carries a labeled
source that persists as a permanent record of how the full text was obtained.

| # | Criterion | Measurable bar |
|---|-----------|----------------|
| AC5 | **Source chip in UI** | For an `include`-verdict study row with a known OA DOI (e.g. any from the validation 002 gold set), clicking "Get full text" renders a chip labeled exactly `via Unpaywall` (or `via OpenAlex`, etc.) with a working href to the resolved URL. A plain unlabeled link is not acceptable. |
| AC5-state | **Loading state** | A spinner or equivalent loading indicator is visible from button click until the API response resolves. No flash of un-styled content; no double-click firing a second request while in-flight. |
| AC5-404 | **404 graceful** | For a study where `/api/fulltext` returns 404 (no OA URL found), the UI shows a static note ("No open-access version found") rather than an error toast, an invisible disabled state, or a crash. |
| AC-persist | **Resolved URL survives reload** | After a resolution + save cycle: reload the screening workbench — the source chip is still rendered (URL and source read from `screening_results`, not client state). Confirm via DevTools Network tab: no `/api/fulltext` call on reload for already-resolved rows. |
| AC-persist-column | **Migration idempotent** | Running the migration SQL twice does not error (`ADD COLUMN IF NOT EXISTS`). Running `npx tsc --noEmit --skipLibCheck` after the schema change passes. |
| AC7 | **No regression** | `npx tsc --noEmit --skipLibCheck` and `npx vitest run` pass with zero new failures after all changes. The 15 pre-existing failures (documented in handoff 083 §8) are unchanged. |

The **beat** is AC5 + AC-persist together: Blindspot surfaces not just a PDF link but a
labeled, persisted source record — an auditable provenance trail that Otto-SR cannot provide
and that will anchor the data-extraction stage's citation chain.

## Notes for the dev

**`ScreeningDecision` fields** (already typed in `types/index.ts` per brief 001):
`fulltext_url?: string`, `fulltext_source?: FulltextSource`, `fulltext_fetched_at?: string`
— verify these exist before modifying `screening/save/route.ts`.

**`FulltextSource` values** (from `types/index.ts`): `"unpaywall" | "openalex" |
"europepmc" | "pmc" | "user_upload"`. The chip label should be the display form:
`Unpaywall`, `OpenAlex`, `Europe PMC`, `PubMed Central`, `Upload`. Map at render time; do
not store display strings in the DB.

**Upsert guard in `screening/save`:** the save route likely upserts on `(search_id, pmid)`
or similar. When adding the fulltext fields, use `COALESCE` or conditional inclusion so a
re-save of a screening verdict (without a fulltext result) does not overwrite a previously
persisted `fulltext_url` with null. Pattern:
```ts
// Only include fulltext fields in the upsert payload when they are present
...(decision.fulltext_url ? {
  fulltext_url: decision.fulltext_url,
  fulltext_source: decision.fulltext_source,
  fulltext_fetched_at: decision.fulltext_fetched_at,
} : {}),
```

**Hydration in `ResultsDashboard`:** when the workbench loads, rows are populated from
`screening_results`. If `fulltext_url` is non-null on the row data, skip the "Get full text"
button entirely and render the source chip directly. This means no extra API call on page
load for already-resolved studies.

**`ResultsDashboard.tsx` is ~3 000 lines** (noted in Deferred non-blockers). Make surgical
edits — do not refactor the component as part of this brief. Locate the `include`-verdict row
render path and add the button/chip there with minimal surrounding change.

**Validation handoff:** after shipping, the tester will verify AC5 (chip renders correctly),
AC-persist (chip survives reload), and AC7 (no regression) in validation 004. The AC2 live
harness (`003-harness.mjs`) does not need re-running — the fanout logic is unchanged.
