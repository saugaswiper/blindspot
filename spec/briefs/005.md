# Brief 005 — Search Recall: Relevance Sort + Comprehensive Pagination

**Date:** 2026-06-23
**Stage in pipeline:** Stage 2 — Literature search (primary-literature capture)
**Status:** UNBLOCKED — no ops/secrets dependency (CRIT-1 does not block this fix)

---

## Goal

Fix `fetchPrimaryStudyIds` across PubMed, EuropePMC, and OpenAlex to use
relevance-sorted, paginated retrieval up to a 2 000-record ceiling, replacing
the current 200-record date-sorted cap that caused the measured 0% recall in
validation 004. Commit the Mitchell 2012 truth set as the first real recall
fixture so future re-tests are deterministic.

---

## Why now

Validation 004 (2026-06-23) produced the **first measured Stage 2 recall:
0/5 = 0%** against the Mitchell 2012 external truth set (5 RCTs with PMIDs
16804151, 15451764, 10086433, 16785771, 1888345). The roadmap flags this as
🔴 FAIL. Per planner rules a measured FAIL on a shipped stage outranks all new
features — brief 004 (upload fallback) is deferred until this is fixed.

Root cause RC1 (primary): both PubMed and EuropePMC return results
newest-first by default; `fetchPrimaryStudyIds` caps at 200 records. For
"cognitive behavioral therapy insomnia NOT systematic[sb]", PubMed has 313
total results; the 5 truth papers sit at positions 201–313 and are never
fetched. EuropePMC has 1 994 total; the truth papers are similarly buried.

Root cause RC2 (terminology variation — 2/5 papers not findable via CBT MeSH):
deferred to brief 006. Fixing RC1 alone moves the theoretical ceiling from
0% → ≥60% (3/5 that ARE reachable via relevance sort) and to ≥100% (5/5) once
synonym expansion follows in 006.

The roadmap's stated differentiator is "higher recall *and* auditable
provenance" vs. Otto-SR. A measured 0% recall invalidates that claim. This is
the minimum fix before any new pipeline stage.

## Acceptance criteria — beat, don't match

Otto-SR claims recall coverage but publishes no per-source breakdown and no
reproducible truth set. The bar: Blindspot ships **measurable, reproducible
recall figures against a public SR ground truth**, with every source labeled and
paginated transparently.

| # | Criterion | Measurable bar |
|---|-----------|----------------|
| AC-pubmed | **PubMed fetches full result set** | `fetchPrimaryStudyIds("cognitive behavioral therapy insomnia")` (no minYear, limit=2000) returns ≥313 PMIDs — the complete result count that PubMed's esearch reports for `... | AC-europepmc | **EuropePMC fetches ≥1 000 records** | `fetchPrimaryStudyIds` on the same query returns ≥1 000 records (page 1 of paginated results). Running the harness: EuropePMC `hits/truth` goes from 0/5 to ≥2/5. |

| AC-fixture | **Real fixture committed** | `REAL_RECALL_FIXTURES` contains exactly one `RecallFixture` entry: `name: "mitchell-2012-cbt-insomnia"`, `synthetic: false`, `truthSet` has 5 entries (PMIDs 16804151, 15451764, 10086433, 16785771, 1888345), `perSource` has non-empty arrays for PubMed and EuropePMC (captured via the capture script after the fixes are in place). |

## Notes for the dev

**PubMed sort param**: PubMed ESearch accepts `sort=relevance` to use the
"Best Match" ML-ranking algorithm (same as the web interface default). The
other callers of `esearch()` (count functions with retmax=1, existing-review
fetches with retmax=50) are unaffected by adding an optional third param —
they will continue to use date sort implicitly by not passing the arg.

**EuropePMC sort**: the parameter name is `sort` and accepted values include
`RELEVANCE`. The `cursorMark` approach: request 1 with `cursorMark=*&pageSize=1000`;
the response includes `nextCursorMark`. If `nextCursorMark` is present and
there are more records, request 2 uses `cursorMark={nextCursorMark}&pageSize=1000`.
Alternatively, EuropePMC supports `page=2&pageSize=1000` if `nextCursorMark`
is absent from the response schema version in use. Inspect the first live
response to confirm which pagination tokens are present; fall back to page-number
if necessary.

**OpenAlex cursor pattern**: the function immediately below `fetchPrimaryStudyIds`
in `lib/openalex.ts` (starting ~line 234, `fetchPrimaryStudiesForScreening`)
already implements cursor-based pagination. Model the `fetchPrimaryStudyIds`
loop on that code.

**Capture script invocation** (for populating the fixture):
```bash
echo "16804151\n15451764\n10086433\n16785771\n1888345" > /tmp/mitchell-truth.txt
npx tsx scripts/capture-recall-fixture.ts \
  --name "mitchell-2012-cbt-insomnia" \
  --description "Mitchell MD et al. BMC Fam Pract 2012;13:40 (PMID 22631616); 5 RCTs from Table 1 (CBT-I vs pharmacotherapy)" \
  --query "cognitive behavioral therapy insomnia" \
  --truth-file /tmp/mitchell-truth.txt \
  --limit 2000 \
  --out lib/fixtures/captured/mitchell-2012.json
```
Run this **after** the three source-file fixes compile cleanly (so you capture
the improved results, not the broken 200-record date-sorted output). Paste the
emitted JSON into `REAL_RECALL_FIXTURES` in `recall-fixtures.ts`.
